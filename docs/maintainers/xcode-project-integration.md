# Xcode Project Integration

This document captures the first supported Xcode integration path for BriteLog.

## Why The First Integration Uses A Scheme Pre-Action

BriteLog needs a signal that means more than "a build happened." The real job is to know:

- which app is about to be run for debugging
- which bundle identifier and scheme that run belongs to
- when a fresh debug run request replaces the previous one

Apple's scheme customization docs make pre-actions and post-actions the most direct run-aware hook. Xcode runs those actions only when the matching scheme action is selected, and script actions can receive build settings from a chosen target. That makes a Run pre-action a better fit than a generic build hook for the first integration path.

Relevant Apple docs:

- <https://developer.apple.com/documentation/xcode/customizing-the-build-schemes-for-a-project>
- <https://developer.apple.com/documentation/xcode/running-custom-scripts-during-a-build>
- <https://developer.apple.com/videos/play/wwdc2022/110401/>

## Why The First Integration Does Not Use A Build Plugin As The Primary Trigger

Swift package build-tool plugins are still build-time extensions. Apple documents them as build commands that run before or during the build, inside the plugin sandbox. They are useful for generating metadata or other build artifacts, but they do not naturally own a live "watch this debug run now" session.

So the current split is:

- scheme pre-action: the run-aware trigger
- native app: the entitlement-bearing host that receives the trigger and owns the live watch session
- future build plugin: an optional convenience or metadata helper, not the primary live-session owner

## Required Inputs For A Live Debug-Run Request

The app needs enough information to target unified logs for the app that is about to launch and to distinguish a fresh run from a stale one.

The first contract uses these fields:

- `requestID`
  - a fresh UUID generated by the scheme pre-action
- `submittedAt`
  - an ISO-8601 UTC timestamp generated by the scheme pre-action
- `source`
  - currently `schemePreAction`
- `projectPath`
  - the `.xcodeproj` path from the scheme context
- `schemeName`
  - the specific scheme that is being run
- `targetName`
  - the target providing build settings for the script
- `bundleIdentifier`
  - the app bundle identifier reported by Xcode build settings
- `buildConfiguration`
  - usually `Debug`
- `builtProductPath`
  - the path to the built app product when Xcode exposes it

That information is enough for the app to:

- know which debug session request is newest
- match `NSWorkspace` launch and terminate events by bundle identifier
- prepare a future local-store watch session around the correct target app

## Trigger File Contract

The first integration writes an incoming request file into the app's Application Support directory:

- directory:
  - `~/Library/Application Support/com.gaelic-ghost.BriteLog/`
- incoming trigger file:
  - `incoming-run-request.env`
- persisted current request:
  - `current-run-request.json`

The scheme pre-action writes the incoming trigger file as line-based key/value pairs. Values that may contain spaces or shell-sensitive characters are base64 encoded and stored with `_b64` suffixes.

Example shape:

```text
requestID=8F0D6B6F-9A1D-4A0F-8A8F-45B2A4B830B1
submittedAt=2026-04-22T23:15:04Z
source=schemePreAction
projectPath_b64=L1VzZXJzL2dhbGV3L1dvcmtzcGFjZS9FeGFtcGxlL0V4YW1wbGUueGNvZGVwcm9q
schemeName_b64=RXhhbXBsZUFwcA==
targetName_b64=RXhhbXBsZUFwcA==
bundleIdentifier_b64=Y29tLmV4YW1wbGUuRXhhbXBsZUFwcA==
buildConfiguration_b64=RGVidWc=
builtProductPath_b64=L1VzZXJzL2dhbGV3L0xpYnJhcnkvRGV2ZWxvcGVyL1hjb2RlL0Rlcml2ZWREYXRhL0V4YW1wbGUvQnVpbGQvUHJvZHVjdHMvRGVidWcvRXhhbXBsZUFwcC5hcHA=
```

The app is responsible for:

- polling for that incoming trigger file
- parsing it into the durable `BriteLogRunRequest` model
- persisting the current request as JSON
- removing the consumed incoming trigger file

## Shared-Scheme Requirement

The first installer path only edits shared `.xcscheme` files.

That means:

- supported source path:
  - `<Project>.xcodeproj/xcshareddata/xcschemes/<Scheme>.xcscheme`
- not yet supported:
  - user-private schemes under `xcuserdata`
- if a project does not have a shared scheme:
  - BriteLog should fail clearly and tell the user to share the scheme in Xcode first

This keeps the first automation path deterministic and avoids mutating developer-private Xcode user data.

## What The Installer Writes Into The Scheme

The installer adds or updates one Launch pre-action with this operator-facing title:

- `Use BriteLog For Debug Runs`

The action:

- provides build settings from the scheme's runnable target
- writes the trigger file into BriteLog's Application Support directory
- asks Launch Services to activate `BriteLog.app` by bundle identifier

The installer must preserve any other existing pre-actions on the scheme.

## App-Side Runtime Responsibilities

Once the request lands, `BriteLog.app` owns the live session boundary.

The app should:

- keep the latest `BriteLogRunRequest`
- track the currently targeted bundle identifier
- watch `NSWorkspace.didLaunchApplicationNotification`
- watch `NSWorkspace.didTerminateApplicationNotification`
- match events against the targeted bundle identifier

Relevant Apple docs:

- <https://developer.apple.com/documentation/appkit/nsworkspace>
- <https://developer.apple.com/documentation/appkit/nsworkspace/didlaunchapplicationnotification>
- <https://developer.apple.com/documentation/appkit/nsworkspace/didterminateapplicationnotification>

This gives the app a real "a new debug run is starting" signal from Xcode plus a real "the app actually launched or exited" signal from macOS.

## Deferred And Future Work

The first project-integration implementation intentionally leaves several things for later:

- a build-phase fallback install option
- a build-plugin convenience path
- automatic sharing or creation of missing shared schemes
- workspace-level scheme editing beyond direct `.xcodeproj` support
- a polished viewer that immediately starts showing local-store output for the matched launch

Those are follow-up steps after the scheme-pre-action contract proves itself.
